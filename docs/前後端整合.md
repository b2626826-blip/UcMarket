# 前後端整合分析報告

> 產出日期：2026-07-08
> 分析方式：後端架構師與前端開發者兩位角色分別獨立掃描程式碼，逐檔比對 `frontend/src/api/*.js` ↔ `backend/.../controller/*` 契約後彙整。
> 標記慣例：`[雙方確認]` 代表前後端兩份獨立分析都指出同一問題，可信度最高。
> 嚴重度：**P0** 阻斷／**P1** 高／**P2** 中／**P3** 低。

---

## 0. 一句話總結

**後端內部（金流鎖、冪等鍵、交易一致性）設計紮實，但前後端「串接層」大面積未完成或串錯**：產品主流程（首頁市場、市場詳情、下注、錢包、持倉、交易紀錄、Email 登入）目前幾乎 100% 使用前端寫死的假資料，對應的 API 函式雖已定義卻從未被呼叫；少數真的有呼叫後端的地方，又存在端點不存在、HTTP 方法錯誤、欄位結構對不上等契約缺陷。

## 2. P0 阻斷級問題（必須先修）

### 2.1 `[雙方確認]` 產品主流程全是前端寫死假資料，核心 API 零呼叫
整個平台最核心的頁面都沒有接後端，對應 API 函式定義了卻無任何呼叫端：

| 頁面 | 檔案 | 狀況 | 未被呼叫的 API |
|---|---|---|---|
| 首頁市場列表 | `pages/public/home/index.jsx:14-22` | 寫死陣列 + `setInterval` 亂數假波動 | `marketApi.getMarkets()` |
| 市場詳情 | `pages/public/market-detail/index.jsx:5-69` | 標題/YES-NO 價格/成交量寫死 | `marketApi.getMarketDetail()` |
| 下注面板 | `components/.../TradePanel.jsx:38-52` | 只做前端驗證 + 假成功動畫 | `tradeApi.createTrade()` |
| 錢包 | `pages/member/wallet/index.jsx:75-92` | 餘額寫死 `$125,430.00` | `walletApi` / `walletStore` |
| 持倉 | `pages/member/positions/index.jsx:2-5` | 寫死陣列 | `positionApi` |
| 交易紀錄 | `pages/member/trade-history/index.jsx:3-7` | 寫死陣列 | `tradeApi` |

風險：極易被誤判「功能已完成可上線驗收」，實際使用者看到的餘額/持倉/盈虧全是假的。

### 2.2 `[雙方確認]` Email 登入／註冊頁表單是假的，且正是 token 過期後的導向頁
- `pages/public/login/LoginPage.jsx:13-20`：`handleSubmit` 只 `setShowToast(true)`，**沒有呼叫 `authStore.login()`**；Email/密碼 `<input>` 是非受控元件（`:73,85-88` 無 `value`/`onChange`），使用者輸入完全不被讀取。`RegisterPage.jsx:14-21` 同樣。
- 只有 Firebase OAuth 按鈕是真的接了 API。
- **致命連鎖**：`client.js:35` 在 401/403 時把使用者導向的正是這個假登入頁 → session 過期後被丟到「按登入只跳假 toast」的死路，無法重新登入。
- 技術債：同產品有兩套登入 UI（`AuthModal` 真、`LoginPage` 假），極危險。

### 2.3 `[雙方確認]` 前端呼叫了後端不存在的端點（404 / 405）
| 前端呼叫 | 檔案 | 後端實況 |
|---|---|---|
| `GET /api/positions` | `positionApi.js:4` | `PositionController` 無 root 路徑 → 404 |
| `GET /api/positions/{id}` | `positionApi.js:8` | 無 `@GetMapping("/{id}")` → 404 |
| `GET /api/trades` | `tradeApi.js:4` | `TradeController` 只有 POST → 405 |

後端實際只有：`/api/positions/me`、`/me/open`、`/market/{marketId}`、`/market/{marketId}/open`。
修法：前端改打 `/me` 系列，或後端補端點（補 `/{id}` 需驗 ownership 防 IDOR）。

### 2.4 `[雙方確認]` `positionApi` 假設欄位與後端 DTO 結構性不符
- 前端假設（`types/position.js:2`）：`{ side, shares, avgPrice, currentPrice, profit }`（單邊持倉模型）。
- 後端實際（`PositionResponse.java:10-19`）：`{ yesShares, noShares, yesCost, noCost, status, updatedAt }`（AMM 雙邊模型）。
- 影響：即使把 2.3 的 404 修好，資料轉換層仍需**整個重寫**，是隱藏成本最高的一項。市場詳情頁假設的 `market.yesPrice/noPrice` 後端 `Market` entity 也不存在（後端只有 `yesPool/noPool`，價格須經 `/api/markets/{id}/odds` 計算）。

### 2.5 `[雙方確認]` `AuthGuard` 無 loading gate，刷新受保護頁 = 被強制登出
- `AuthGuard.jsx:4-7` 只看 `user`，不看是否已完成初次驗證。
- `UserLayout.jsx:18-20` 在 mount 才非同步觸發 `checkAuth()`。
- 刷新 `/portfolio` 等頁時，`AuthGuard` 首次 render 時 `user` 必為初始 `null` → 立即 `<Navigate to="/" />`，即使 JWT 仍有效也被踢出。
- 修法：`authStore` 增 `authChecked` boolean，`AuthGuard` 在 `false` 時顯示 loading，不做導向。

### 2.6 `[雙方確認]` `client.js` `BASE_URL` 寫死 `http://localhost:8080`
- `client.js:1` 硬編碼，未讀 `import.meta.env.VITE_API_BASE_URL`；`.env.example` 也只有 Firebase 四把 key。
- 影響：production build 會打到本機，等同正式環境完全打不通後端。
- 修法：改用環境變數並補進 `.env.example`。

## 3. P1 高風險問題

### 3.1 `[雙方確認]` 登出 API 方法錯誤 + refresh token 完全未落地
- `authApi.logout()`（`authApi.js:15-17`）用 **GET** 呼叫，後端 `AuthController.java:69-76` 要求 **POST + `@RequestBody LogoutRequest{refreshToken}`** → 必失敗（405/400）。
- 前端 `authStore.js:65-68` 把錯誤吞掉，登出「看起來成功」，但後端 refresh token 從未被 `revoke()`（`AuthService.java:90-99`）→ 撤銷不了的殭屍 session，直到 7 天自然過期。
- 前端從未儲存/使用 `refreshToken`（全專案 grep 僅後端出現），`AuthResponse.refreshToken`/`expiresIn` 是死資料；`client.js` 對 401/403 只清 token 跳轉，**無靜默續期** → session 壽命 = access token TTL，中途硬中斷。
- 修法：logout 改 POST+body；落地 refresh 流程或改為短命 access token 單 token 設計。

### 3.2 `[後端]` `/api/positions/market/{marketId}` 無擁有者檢查（IDOR）
- `PositionController.java:36-44` 直接吃 `marketId`，`PositionService.getPositionsByMarketId`（`:53-61`）回傳該市場**所有使用者**的 `PositionResponse`（含 `userId`、`yesCost`、`noCost`、`yesShares`、`noShares`）。
- `SecurityConfig` 僅 `/api/admin/**` 需 ADMIN，這兩支落在 `authenticated()` → 一般會員可互查對方持倉成本。
- 與上一 commit `5bcffd0`（`/me` 防 IDOR）同類，但 `/market/{marketId}` 系列尚未修。
- 修法：改聚合 DTO（不含 userId，只給市場總量）或限 ADMIN。

### 3.3 `[後端]` `GlobalExceptionHandler` 未攔 Spring 內建例外
- `GlobalExceptionHandler.java` 缺 405（`HttpRequestMethodNotSupportedException`，正是 3.1 logout 觸發的）、壞 JSON body（`HttpMessageNotReadableException`）等，會落回 Spring 預設格式而非 `{timestamp,status,error,message}`。
- 修法：補齊常見 Spring 例外 handler，統一錯誤格式。

### 3.4 `[前端]` 已建 store / API 與畫面完全脫節（死代碼）
- `walletStore.fetchWallet()` 呼叫真實 `getWallet()`，但 `WalletPage` 從未 import 該 store，餘額為寫死字串。
- `TradePanel` 與 `walletStore` 毫無串接、無樂觀更新（因交易根本沒送出）。
- `PortfolioPage`（`portfolio/index.jsx:8`）重複呼叫 `checkAuth()`（與 `UserLayout` 各一次），卻只用來取 `username`，餘額/損益仍寫死。

### 3.5 `[前端]` 通用 catch 吞錯，使用者看不到失敗原因
- `walletStore.js:14-16`、`authStore.js:17-20` 的 catch 只降級成預設值（餘額 0 / 未登入），不記錄不展示 → 網路暫斷或後端 500 時，使用者無法分辨「真的是 0」還是「請求失敗」。

### 3.6 `[前端]` `types/*.js` 無任何 runtime 驗證
- 全是註解或常數物件，無 zod/io-ts。後端改欄位名不會有編譯期/執行期警訊，只會在畫面悄悄變 `undefined`。

## 5. P3 低風險 / 待確認

- **`[前端]` 登出無跨 store 清理**：`authStore.logout()` 不重置 `walletStore`，日後接上後同瀏覽器換帳號會看到餘額殘影。（`authStore.js:65-71`）
- **`[後端]` `DevWalletController` 單層防線**：僅靠 `@Profile("dev")`，dev 環境任何登入用戶可對任意 `userId` 印錢/扣款，建議加 `hasRole("ADMIN")`。（`DevWalletController.java:16-19`）
- **`[後端]` `LocalDateTime` 無時區**：`createdAt`/`closeAt` 等序列化不含時區，前端 `new Date()` 當本地時區解析，跨時區會偏移。
- **`[後端]` 錯誤訊息中英夾雜**：`handleAuth`/`handleForbidden` 固定英文，業務例外多為中文。（`GlobalExceptionHandler.java:69-77`）
- **`[後端]` 重複定義**：`tradeApi.js:11-13` 與 `adminApi.js:29-32` 各有一份 `getAdminTransactions()`，契約漂移風險。
- **`[前端]` 型別文件脫節**：`types/position.js`、`types/wallet.js`（`balance` vs 實際 `balanceAfter`）與後端 DTO 不符。
- **`[待確認]` Firebase OAuth 失敗路徑**：`firebaseLogin`（`AuthModal.jsx:108-133`）呼叫正確，但後端驗證失敗時前端錯誤訊息品質未驗，需實跑一次失敗情境。

## 6. 建議修復優先序（合併雙方 Top 5）

> 順序原則：先讓「已登入使用者不被卡死」，再讓「主流程真的接上後端」，最後補契約與安全。

1. **P0 修好登入死循環**：`/auth/login`、`/auth/register` 表單接上 `authStore.login()`/`register()`（或直接改用 `AuthModal` 的實作，移除假登入頁）。這是唯一會讓正常使用者完全卡死的問題。（2.2）
2. **P0 修 `AuthGuard` loading gate**：加 `authChecked`，避免刷新即被登出，影響每一位登入者。（2.5）
3. **P0 修端點與環境變數**：修正 `positions`/`trades` 三個 404/405 端點路徑、`BASE_URL` 改環境變數。（2.3、2.6）
4. **P1 修登出與 refresh 契約**：logout 改 POST+body；決定落地 refresh 流程或改單 token；補 `GlobalExceptionHandler` 內建例外。（3.1、3.3）
5. **P0/P1 串接主流程並重寫持倉資料模型**：把首頁市場、市場詳情、下注、錢包、持倉、交易紀錄接上真實 API，並依後端 AMM 雙邊模型（`yesShares/noShares/yesCost/noCost`）重寫前端資料轉換，順帶收緊 `/api/positions/market/{marketId}` 授權（IDOR）。（2.1、2.4、3.2）
