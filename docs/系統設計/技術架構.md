# UcMarket 技術架構

> 文件定位：架構原則與目標設計。現行技術版本、已接通模組與未實作範圍請見 `../current-implementation.md`；通知工作、SELL 與多選項市場仍是後續規劃。第一階段自動化採 Java／Spring Boot，不導入 n8n。

## 1. 專案定位

UcMarket 是以「前後端分離」為核心設計的 Java 專題作品。專案目標不只是完成基本功能，而是透過完整的系統架構展現 Java 後端開發、RESTful API 設計、資料庫設計、前端整合，以及 Spring 排程與可靠通知工作設計能力。

本專案預計作為資策會跨域 Java 工程師課程的期末專題與就業作品集，架構設計會兼顧課程所學、實務可維護性與面試展示價值。

## 3. Core Domain 核心領域

Core Domain 是 UcMarket 的核心業務層，描述系統真正要解決的產品問題。技術架構負責支撐系統運作，但專案的重點會集中在市場、交易、錢包、持倉、結算與排行榜等核心流程。

| 核心領域 | 職責 |
|---|---|
| Market | 市場建立、審核、狀態管理、截止時間、結果設定與市場類型 |
| Trade | Yes / No 或市場選項買入賣出、交易紀錄、價格或比例計算 |
| Wallet | 點數餘額、儲值、扣款、退款與錢包異動紀錄 |
| Position | 使用者持倉、持倉數量、平均成本與可結算狀態 |
| Resolution | 市場結果確認、結算流程、獎勵發放與防止重複結算 |
| Ranking | 熱門市場、交易量、使用者活躍度與排行榜資料 |

這一層不一定需要在專案根目錄建立獨立資料夾，而是作為系統設計與功能切分的核心依據。實作時會由 Backend 的 Service、Entity、Repository 與 Database tables 共同支撐。

## 4. 前後端分離設計

本專案採用前後端分離架構，前端負責使用者介面與互動，後端負責商業邏輯、資料庫操作、權限控管與 API 服務。

```text
Frontend
React + JavaScript
        ↓
HTTP Request
Axios / Fetch
        ↓
Backend
Spring Boot RESTful API
        ↓
Database
PostgreSQL
```

前端不直接操作資料庫，而是透過後端提供的 API 取得或修改資料。後端則以 JSON 格式回傳資料給前端。

範例流程：

```text
使用者在前端按下買入按鈕
        ↓
前端呼叫後端交易 API
        ↓
Spring Boot 驗證登入狀態與錢包餘額
        ↓
Service 層建立交易紀錄並扣除錢包餘額
        ↓
Repository 寫入資料庫
        ↓
後端回傳交易結果
        ↓
前端更新畫面
```

## 5. 後端分層架構

後端採用 Java、Spring Boot、Spring MVC、Spring Security 與 Spring Data JPA / Hibernate 建立 RESTful API。整體採用常見的 Spring Boot 分層架構，讓每一層職責清楚，方便維護與測試。

```text
Controller
接收前端 request，回傳 response
        ↓
Service
處理商業邏輯，例如交易、錢包扣款、市場結算
        ↓
Repository
操作資料庫，負責查詢與資料存取
        ↓
Entity
對應資料庫資料表
```

### 各層職責

| 層級 | 職責 |
|---|---|
| Controller | 定義 API 路由，接收 request，回傳 response |
| Service | 處理核心商業邏輯，例如交易、錢包、結算 |
| Repository | 負責資料庫存取 |
| Entity | 對應資料表結構 |
| DTO | 作為前後端資料傳輸格式，避免直接暴露 Entity |
| Security | 處理登入驗證與權限控管 |
| Exception | 統一處理錯誤回應 |
| Config | 管理專案設定 |

## 9. 自動化在專案中的角色

第一階段自動化直接放在現有 Spring Boot 程序中，使用 `@Scheduled` 與資料庫持久化工作，不新增 n8n、Kafka、RabbitMQ、Redis 或獨立 Worker。核心交易、錢包、持倉與結算仍由 Service 層的 transaction 管理。

目前已存在四條排程：

| 工作 | 頻率 | 實作位置 |
|---|---|---|
| 關閉逾期市場 | 每 60 秒 | `MarketService` |
| 價格歷史 snapshot | 每 5 分鐘 | `PriceHistoryService` |
| 建立天氣市場 | 應用程式啟動時、每日 12:00 | `WeatherMarketService` |
| 天氣市場結算 | 預設每小時 | `WeatherMarketResolutionService` |

通知功能採「同一交易建立通知工作、交易提交後再寄送」的方向，避免寄信成功但資料庫 rollback，或資料成功但通知永久遺失。完整 schema、重試策略與工作包請見 `自動化系統規劃.md`。

## 10. 自動化演進順序

1. 先固定既有送審、交易、結算與天氣排程行為。
2. 建立 notification job 與唯一事件鍵，確保重複執行不重複寄送。
3. 由 Spring worker claim 工作，使用測試用 sender 驗證成功、失敗與重試。
4. 接上正式 Email，再擴充截止提醒、結算通知與每日報表。
5. 只有未來出現跨服務或吞吐需求時，再重新評估外部 workflow／message broker。

## 12. 資料庫設計方向

主要資料表可先規劃如下：

```text
users
roles
markets
market_options
trades
wallets
wallet_transactions
positions
market_reviews
market_price_history
notifications
admin_logs
```

### 資料表職責

| 資料表 | 用途 |
|---|---|
| users | 儲存會員資料 |
| roles | 儲存權限角色 |
| markets | 儲存市場主資料與市場類型，例如 binary、count_range、multiple_choice |
| market_options | 儲存市場選項，例如 Yes / No、0 次、1 次、2-3 次 |
| trades | 儲存交易紀錄 |
| wallets | 儲存會員錢包餘額 |
| wallet_transactions | 儲存錢包異動紀錄 |
| positions | 儲存使用者在各市場選項中的持倉 |
| market_reviews | 儲存市場審核紀錄 |
| market_price_history | 儲存市場價格或比例變化紀錄 |
| notifications | 儲存通知紀錄 |
| admin_logs | 儲存後台操作紀錄 |

### 交易一致性設計

UcMarket 有交易、錢包、持倉與結算，因此資料庫不只是儲存資料，也需要確保交易流程的一致性。建立交易、錢包扣款、持倉更新與交易紀錄寫入，必須在同一個交易流程中完成。

後端 Service 層已在交易、錢包與結算流程使用 `@Transactional`，並持續確保：

- 使用者點數不會扣款成功但交易失敗
- 交易成功後才更新持倉
- 任一步驟失敗時資料可以回復
- 市場結算不能重複發放獎勵
- 錢包異動與交易紀錄可以互相追蹤
- 次數型市場結算時，實際數值只能對應到一個勝出選項

範例流程：

```text
使用者送出交易
        ↓
檢查登入狀態、錢包餘額、市場狀態
        ↓
建立交易紀錄
        ↓
扣除錢包餘額並建立錢包異動紀錄
        ↓
更新使用者持倉
        ↓
交易成功後寫入通知工作；寄送在 transaction commit 後執行（規劃）
```

## 15. 架構設計原則

- 前端負責畫面與互動，後端負責資料與邏輯。
- Core Domain 用來定義市場、交易、錢包、持倉、結算與排行榜等核心業務。
- 核心交易邏輯與第一階段自動化放在 Spring Boot。
- 使用 DTO 作為 API 傳輸格式，避免直接暴露 Entity。
- Service 層集中處理商業邏輯，Controller 保持簡潔。
- 交易、錢包扣款、持倉更新與結算需要確保資料一致性。
- 交易與錢包異動需要保留完整紀錄，方便追蹤與除錯。
- 通知、排程與報表先依 `自動化系統規劃.md` 實作；未達跨服務需求前不新增外部 workflow 工具。
- API 文件、README、架構圖與 Demo 流程都要納入作品集展示。
