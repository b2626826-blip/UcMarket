# 前端架構與撰寫指南

> 現行頁面路由、後端 API 與尚未實作範圍請見 `../../docs/current-implementation.md`。本指南負責前端檔案責任與撰寫規則，不另行定義後端契約。

本文件是 `frontend/` 的單一入口文件，合併原本的資料夾規劃與類別下注頁撰寫指南。內容以目前程式碼為準，用於確認檔案責任、路由位置與開發方式。

## 1. 技術定位

`frontend/` 是正式前端，現況使用：

- React + Vite
- React Router
- Zustand
- Chart.js
- Bootstrap / Bootstrap Icons
- Firebase Authentication

前端負責畫面、互動、表單驗證、路由、登入狀態與 REST API 呼叫；資料與交易結果仍以後端為準，前端不直接連接資料庫，也不自行決定成交或結算結果。

## 2. 快速開始

```bash
cd frontend
npm install      # 第一次或依賴異動後執行
npm run dev      # 啟動開發伺服器
npm run build    # 提交前確認可編譯
```

OAuth 登入前，先將 `.env.example` 複製為 `.env`，再填入 `VITE_FIREBASE_API_KEY`、`VITE_FIREBASE_AUTH_DOMAIN`、`VITE_FIREBASE_PROJECT_ID` 與 `VITE_FIREBASE_APP_ID`。天氣頁若要讀取中央氣象署正式資料，另設定 `VITE_CWA_API_KEY`；未設定時會使用頁面內的 mock 預報。`.env` 不可提交到 Git。

## 4. 各資料夾責任

### `src/pages`

放完整頁面與路由入口。頁面負責組合資料、共用元件與頁面流程，不應重複實作共用版型或把 API 細節散落在各頁。

- `public/`：首頁、登入、註冊、市場列表與市場詳情。
- `member/`：建立市場、資產、錢包、持倉、交易紀錄、排行榜與個人資料。
- `admin/`：後台登入、儀表板、市場、使用者、管理員、交易、設定與操作紀錄。

### `src/components`

放真正會重複使用的 UI 元件。

- `layout/`：`UserLayout`、`AdminLayout`、Navbar、Footer。
- `common/`：Button、Input、Modal、Loading、Toast、`DetailPageTemplate` 等。
- `finance/`：金融詳情頁使用的 TradingView 圖表與新聞 widget。
- `market/`：市場卡、標題、統計資訊與 `TradePanel`。
- `admin/`：後台表格與確認對話框。

只使用一次的畫面先留在頁面內，不為了拆分而建立抽象。

### `src/api`

集中管理後端 API 呼叫：

| 檔案 | 責任 |
|---|---|
| `client.js` | base URL、認證標頭、HTTP 方法、回應與錯誤處理 |
| `authApi.js` | 登入與身分驗證 |
| `oauthApi.js` | 將 Firebase ID token 送到後端 OAuth 登入端點 |
| `marketApi.js` | 市場資料 |
| `tradeApi.js` | 交易試算與建立交易 |
| `walletApi.js` | 錢包與異動紀錄 |
| `positionApi.js` | 持倉資料 |
| `rankingApi.js` | 排行榜資料 |
| `adminApi.js` | 後台操作 |

頁面不要另外硬寫 base URL 或重複建立通用 fetch 邏輯。API 回傳格式與錯誤訊息以後端契約為準。

目前有兩個已知契約缺口：`marketApi.getMarketPriceHistory()` 與 `tradeApi.getTrades()` 已宣告前端呼叫，但後端尚未提供對應 GET route。完成後端端點前，頁面不可把這兩項當成已接通功能。

### `src/router`

- `routes.jsx`：所有路由與 layout 階層。
- `AuthGuard.jsx`：限制會員頁面。
- `AdminGuard.jsx`：限制管理員頁面；目前開發模式允許未登入進入後台，正式部署前需重新確認此行為。

目前主要路由：

| 路由 | 頁面／權限 |
|---|---|
| `/` | Landing page |
| `/home`、`/markets` | 首頁／市場列表 |
| `/markets/current-affairs` | 重導回首頁；時事市場已整合進首頁 |
| `/markets/politics` | 政治市場列表 |
| `/markets/weather` | 天氣市場列表 |
| `/markets/weather/:id` | 天氣市場詳情 |
| `/markets/politics/:id` | 政治市場詳情 |
| `/markets/sports/:id` | 運動市場詳情 |
| `/markets/current-affairs/:id` | 時事市場詳情 |
| `/markets/finance/:id` | 金融市場詳情 |
| `/auth/login`、`/auth/register` | 登入、註冊 |
| `/rankings` | 排行榜，目前是公開路由 |
| `/portfolio`、`/wallet`、`/positions`、`/trades` | 會員路由 |
| `/markets/new` | 會員建立市場 |
| `/profile` | 會員個人資料、頭像、簡介與修改密碼 |
| `/admin/login` | 後台登入 |
| `/admin/*` | 管理員後台 |

新增或調整頁面時，必須同步檢查 `routes.jsx` 的 import、path、layout 與 guard。

### `src/store`

Zustand store 只放跨頁共享狀態：

- `authStore.js`：使用者、一般登入、Firebase OAuth 登入狀態與角色。
- `walletStore.js`：錢包摘要與餘額。
- `uiStore.js`：全站通知與 UI 狀態。

單一頁面的表單或暫存資料留在頁面 state，不放進全域 store。

### `src/types`

目前以 JavaScript 檔記錄 auth、market、trade、wallet、position、ranking、admin 等資料形狀。欄位名稱應與實際 API 契約一致，不在前端自行創造另一套命名。

### `src/assets`、`config`、`hooks`、`mocks`

- `assets/styles/theme.css`：設計 token 與主題樣式。
- `assets/styles/base.css`：reset、全站 layout、共用 class 與 CSS 變數。
- `config/firebase.js`：由 Vite 環境變數初始化 Firebase，並提供 Google、GitHub 與 Facebook provider。目前登入與註冊頁有接上 Google、GitHub。
- `hooks/useGlowEffect.js`：追蹤滑鼠位置並提供卡片光暈座標。
- `mocks/`：尚未接 API 時使用的明確 mock 資料，不應偽裝成正式 API 回傳。

## 5. 類別市場詳情頁分工

每個類別的入口檔如下：

| 類別 | 檔案 | 路由 |
|---|---|---|
| 天氣 | `src/pages/public/market-detail-weather/index.jsx` | `/markets/weather/:id` |
| 政治 | `src/pages/public/market-detail-politics/index.jsx` | `/markets/politics/:id` |
| 運動 | `src/pages/public/market-detail-sports/index.jsx` | `/markets/sports/:id` |
| 時事 | `src/pages/public/market-detail-current-affairs/index.jsx` | `/markets/current-affairs/:id` |
| 金融 | `src/pages/public/market-detail-finance/index.jsx` | `/markets/finance/:id` |

多人平行開發時，每位組員只修改自己負責的類別資料夾；若需要改 `DetailPageTemplate`、`TradePanel`、全域 CSS 或路由，先確認這是所有類別共同需求，避免影響其他頁面。

## 6. `DetailPageTemplate` 使用方式

類別詳情頁由 `components/common/DetailPageTemplate.jsx` 提供共用骨架：

- Navbar 與 Footer 由外層 `UserLayout` 提供。
- Hero 顯示市場 ID、副標與可選的額外內容。
- 主區域是左側 `children` 與右側 `TradePanel`。
- 小螢幕的欄位堆疊由共用樣式處理。
- `.trade-market-card` 與 `.trade-panel` 已有金邊、升浮與追光效果。

Props：

| Prop | 型別 | 必填 | 說明 |
|---|---|---|---|
| `id` | `string \| number` | 是 | `useParams()` 取得的市場 ID |
| `subtitle` | `string` | 是 | Hero 副標 |
| `marketId` | `string \| number` | 否 | 傳給 `TradePanel`，未提供時使用 `id` |
| `category` | `string` | 否 | 自訂 Hero 標題；未提供時顯示市場 ID |
| `heroLayout` | `'left' \| 'full'` | 否 | Hero 放在左欄或橫跨內容上方，預設為 `left` |
| `heroExtras` | `ReactNode` | 否 | Hero 右側額外內容 |
| `startTime` | `string` | 否 | Hero 的開始時間 |
| `settleTime` | `string` | 否 | Hero 的結算時間 |
| `status` | `string` | 否 | Hero 的市場狀態 |
| `settlementRule` | `string` | 否 | Hero 的結算規則 |
| `market` | `object` | 否 | 傳給預設 `TradePanel` 的市場價格資料 |
| `tradePanel` | `ReactNode` | 否 | 取代預設 `TradePanel` |
| `children` | `ReactNode` | 否 | 左欄市場資訊、圖表與相關內容 |

最小範例：

```jsx
import { useParams } from 'react-router-dom';
import DetailPageTemplate from '../../../components/common/DetailPageTemplate';

export default function PoliticsDetailPage() {
  const { id } = useParams();

  return (
    <DetailPageTemplate id={id} subtitle="政治事件預測市場" marketId={id}>
      <div className="trade-market-card">
        {/* 此類別的市場資訊與圖表 */}
      </div>
    </DetailPageTemplate>
  );
}
```

### 天氣詳情頁

`market-detail-weather/` 目前包含天氣門檻盤口、價格走勢圖、天氣預報圖表、區域切換與 modal。CWA 預報讀取集中在同資料夾的 `weatherApi.js`；根層 `src/api/weatherApi.js` 已移除，不要再建立第二份相同邏輯。市場本身仍透過 `src/api/marketApi.js` 讀取後端資料。

## 7. 樣式與特效規則

### 優先重用現有樣式

若 `.trade-market-card` 等共用 class 已符合需求，不要建立重複規則。類別頁需要大量專屬樣式時，才在同一資料夾新增 CSS 並由 `index.jsx` 引入：

```text
market-detail-politics/
├── index.jsx
└── PoliticsDetailPage.css
```

類別專屬 class 使用前綴，避免污染其他頁面：

- `.weather-`
- `.politics-`
- `.sports-`
- `.current-affairs-`
- `.finance-`

小量、只出現一次的動態值可使用 inline style；可重用或涉及 RWD、hover 的規則應放在 CSS。

### 全域變數

實際可用變數定義在 `base.css` 與 `theme.css`，常用值如下：

```css
var(--bg)             /* #040404 */
var(--card)           /* #090909 */
var(--gold)           /* #d9aa43 */
var(--gold-soft)      /* #b88b2f */
var(--green)          /* #00d66f */
var(--red)            /* #ff476d */
var(--text)           /* #ffffff */
var(--text-secondary) /* #8f8f8f */
var(--border)         /* rgba(255,255,255,.06) */
```

不要在類別頁重複宣告同名全域變數。

### 自訂卡片追光

只有自訂卡片也需要追光時，才引入 hook：

```jsx
import useGlowEffect from '../../../hooks/useGlowEffect';

useGlowEffect('.politics-chart-card');
```

對應 CSS 必須提供定位、裁切與 `::after` 光暈；若只需要 hover 升浮而不需要追光，不必使用 hook。

### RWD

共用詳情頁在 `1200px` 以下會處理左右欄堆疊。類別內部的圖表或卡片仍需依自身結構補 media query，例如：

```css
@media (max-width: 992px) {
  .politics-chart-grid {
    grid-template-columns: 1fr;
  }
}
```

## 8. 開發邊界

- 優先修改負責頁面，不順手重構共用元件。
- 共用元件只放多個頁面確實共用的內容。
- 後端權限檢查不可只依賴前端 guard。
- 錢包、價格、成交、持倉與結算資料以後端回傳為準。
- 審核、結算、停權等高影響操作需有確認流程，完成後重新讀取後端狀態。
- mock 資料必須集中且明確標示，接上 API 後移除對應 mock 路徑。
- 不新增未被需求使用的頁面、設定或抽象層。

## 9. 完成檢查

每次修改至少確認：

1. 目標路由能開啟，且 layout、guard 符合頁面權限。
2. 瀏覽器 console 無新增錯誤。
3. 桌面與手機寬度沒有水平溢位或內容遮擋。
4. API loading、成功、空資料與失敗狀態有合理呈現。
5. 在 `frontend/` 執行 `npm run build` 通過。
6. OAuth 與天氣正式資料需求已設定必要環境變數，且 `.env` 未被 Git 追蹤。

排行榜視覺規範另見 `排行榜設計語彙.md`；時事列表與盤口細節另見 `時事盤口設計規格.md`。
