# UcMarket 技術架構

## 1. 專案定位

UcMarket 是以「前後端分離」為核心設計的 Java 專題作品。專案目標不只是完成基本功能，而是透過完整的系統架構展現 Java 後端開發、RESTful API 設計、資料庫設計、前端整合，以及 n8n 自動化流程應用能力。

本專案預計作為資策會跨域 Java 工程師課程的期末專題與就業作品集，架構設計會兼顧課程所學、實務可維護性與面試展示價值。

## 3. Core Domain 核心領域

Core Domain 是 UcMarket 的核心業務層，描述系統真正要解決的產品問題。技術架構負責支撐系統運作，但專案的重點會集中在市場、交易、錢包、持倉、結算與排行榜等核心流程。

| 核心領域 | 職責 |
|---|---|
| Market | 市場建立、審核、狀態管理、截止時間、結果設定與市場類型 |
| Trade | Yes / No 或市場選項買入賣出、交易紀錄、價格或比例計算 |
| Wallet | 點數餘額、儲值、扣款、退款與錢包異動紀錄 |
| Position | 使用者持倉、持倉數量、平均成本與可結算狀態 |
| Resolution | 市場結果確認、結算流程、獎勵發放與防止重複結算 |
| Ranking | 熱門市場、交易量、使用者活躍度與排行榜資料 |

這一層不一定需要在專案根目錄建立獨立資料夾，而是作為系統設計與功能切分的核心依據。實作時會由 Backend 的 Service、Entity、Repository 與 Database tables 共同支撐。

## 4. 前後端分離設計

本專案採用前後端分離架構，前端負責使用者介面與互動，後端負責商業邏輯、資料庫操作、權限控管與 API 服務。

```text
Frontend
React + JavaScript
        ↓
HTTP Request
Axios / Fetch
        ↓
Backend
Spring Boot RESTful API
        ↓
Database
PostgreSQL
```

前端不直接操作資料庫，而是透過後端提供的 API 取得或修改資料。後端則以 JSON 格式回傳資料給前端。

範例流程：

```text
使用者在前端按下買入按鈕
        ↓
前端呼叫後端交易 API
        ↓
Spring Boot 驗證登入狀態與錢包餘額
        ↓
Service 層建立交易紀錄並扣除錢包餘額
        ↓
Repository 寫入資料庫
        ↓
後端回傳交易結果
        ↓
前端更新畫面
```

## 5. 後端分層架構

後端採用 Java、Spring Boot、Spring MVC、Spring Security 與 Spring Data JPA / Hibernate 建立 RESTful API。整體採用常見的 Spring Boot 分層架構，讓每一層職責清楚，方便維護與測試。

```text
Controller
接收前端 request，回傳 response
        ↓
Service
處理商業邏輯，例如交易、錢包扣款、市場結算
        ↓
Repository
操作資料庫，負責查詢與資料存取
        ↓
Entity
對應資料庫資料表
```

### 各層職責

| 層級 | 職責 |
|---|---|
| Controller | 定義 API 路由，接收 request，回傳 response |
| Service | 處理核心商業邏輯，例如交易、錢包、結算 |
| Repository | 負責資料庫存取 |
| Entity | 對應資料表結構 |
| DTO | 作為前後端資料傳輸格式，避免直接暴露 Entity |
| Security | 處理登入驗證與權限控管 |
| Exception | 統一處理錯誤回應 |
| Config | 管理專案設定 |

## 9. n8n 在專案中的角色

n8n 在本專案中定位為「自動化流程層」，負責通知、排程、報表與外部服務整合，不負責核心交易邏輯。

**n8n 只做通知、排程、報表；核心交易邏輯一定放在 Spring Boot。**

核心商業邏輯仍由 Spring Boot Service 層處理，例如：

- 交易建立
- 錢包扣款
- 市場結算
- 權限驗證
- 資料一致性控管

n8n 適合處理的工作：

- 交易成功後發送通知
- 每日固定時間產生熱門市場報表
- 市場截止前提醒使用者或管理員
- 發生系統錯誤時通知管理員

```text
使用者操作前端
        ↓
Spring Boot 處理交易
        ↓
寫入資料庫
        ↓
呼叫 n8n Webhook
        ↓
n8n 發送通知 / 報表 / 提醒
```

## 12. 資料庫設計方向

主要資料表可先規劃如下：

```text
users
roles
markets
market_options
trades
wallets
wallet_transactions
positions
market_reviews
market_price_history
notifications
admin_logs
```

### 資料表職責

| 資料表 | 用途 |
|---|---|
| users | 儲存會員資料 |
| roles | 儲存權限角色 |
| markets | 儲存市場主資料與市場類型，例如 binary、count_range、multiple_choice |
| market_options | 儲存市場選項，例如 Yes / No、0 次、1 次、2-3 次 |
| trades | 儲存交易紀錄 |
| wallets | 儲存會員錢包餘額 |
| wallet_transactions | 儲存錢包異動紀錄 |
| positions | 儲存使用者在各市場選項中的持倉 |
| market_reviews | 儲存市場審核紀錄 |
| market_price_history | 儲存市場價格或比例變化紀錄 |
| notifications | 儲存通知紀錄 |
| admin_logs | 儲存後台操作紀錄 |

### 交易一致性設計

UcMarket 有交易、錢包、持倉與結算，因此資料庫不只是儲存資料，也需要確保交易流程的一致性。建立交易、錢包扣款、持倉更新與交易紀錄寫入，必須在同一個交易流程中完成。

後端 Service 層未來會使用類似 `@Transactional` 的概念處理，確保：

- 使用者點數不會扣款成功但交易失敗
- 交易成功後才更新持倉
- 任一步驟失敗時資料可以回復
- 市場結算不能重複發放獎勵
- 錢包異動與交易紀錄可以互相追蹤
- 次數型市場結算時，實際數值只能對應到一個勝出選項

範例流程：

```text
使用者送出交易
        ↓
檢查登入狀態、錢包餘額、市場狀態
        ↓
建立交易紀錄
        ↓
扣除錢包餘額並建立錢包異動紀錄
        ↓
更新使用者持倉
        ↓
交易成功後才呼叫 n8n Webhook 發送通知
```

可以用以下方式介紹本專案架構：

> 我的專題採用前後端分離架構，前端負責市場列表、交易畫面、會員與後台管理介面，後端使用 Spring Boot 建立 RESTful API，並透過 Spring Data JPA / Hibernate 操作資料庫。系統核心領域包含市場、交易、錢包、持倉、結算與排行榜，其中交易、錢包扣款與持倉更新會集中在 Service 層以交易一致性的方式處理。除此之外，我整合 n8n 作為自動化流程工具，用來處理交易通知、排程報表與市場截止提醒，但核心交易邏輯仍保留在 Spring Boot。

## 15. 架構設計原則

- 前端負責畫面與互動，後端負責資料與邏輯。
- Core Domain 用來定義市場、交易、錢包、持倉、結算與排行榜等核心業務。
- 核心交易邏輯放在 Spring Boot，不放在 n8n。
- 使用 DTO 作為 API 傳輸格式，避免直接暴露 Entity。
- Service 層集中處理商業邏輯，Controller 保持簡潔。
- 交易、錢包扣款、持倉更新與結算需要確保資料一致性。
- 交易與錢包異動需要保留完整紀錄，方便追蹤與除錯。
- n8n 用於通知、排程、報表與外部服務整合。
- API 文件、README、架構圖與 Demo 流程都要納入作品集展示。
