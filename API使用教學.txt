名稱	API	主要邏輯
查使用者全部持倉	GET /api/positions/user/{userId}	用 userId 到 positions 表查資料，回傳這個使用者所有持倉，不管狀態是 OPEN、SETTLED 或 CANCELED 都會出現。
查使用者 OPEN 持倉	GET /api/positions/user/{userId}/open	用 userId 加上 status = OPEN 查 positions，只回傳尚未結算、還在進行中的持倉。
查全部市場	GET /api/markets	從 markets 表查出所有市場，讓前端可以顯示市場列表。
查單一市場	GET /api/markets/{marketId}	用 marketId 查單一市場。如果找不到，就回 404 Market not found。
啟用市場	POST /api/markets/{marketId}/activate	用 marketId 找市場，找到後把 status 改成 ACTIVE，這樣交易 API 才允許買入。
試算交易	POST /api/markets/{marketId}/trades/quote	先確認市場是 ACTIVE，再依照目前 yesPool/noPool 計算買入價格 price，最後用 amount / price 算出可得到的 shares。這支只試算，不會寫入交易、不會更新持倉、不會改市場池。
正式買入	POST /api/markets/{marketId}/trades/buy?userId={userId}	先確認市場是 ACTIVE，計算 price 和 shares，建立一筆 Trade。接著呼叫 PositionService.addBuyPosition(...)，如果這個使用者在該市場還沒有持倉就新增一筆；如果已有持倉就累加 yesShares/yesCost 或 noShares/noCost。最後更新市場池 yesPool/noPool。
結算市場	POST /api/markets/{marketId}/resolution/resolve?result=YES/NO	先確認市場存在、不是 RESOLVED，且狀態是 ACTIVE 或 CLOSED。接著查出該市場所有 OPEN 持倉，全部改成 SETTLED，最後把市場狀態改成 RESOLVED，並記錄結果 YES 或 NO。
取消市場	POST /api/markets/{marketId}/resolution/cancel	用 marketId 找市場，把市場狀態改成 CANCELED，並把該市場所有 OPEN 持倉改成 CANCELED。
你的持倉核心邏輯可以另外這樣講：
情境	邏輯
第一次買某市場	positions 沒有該 userId + marketId，所以新增一筆持倉，狀態是 OPEN。
重複買同一市場	positions 已有該 userId + marketId，所以更新同一筆，不新增第二筆。
買 YES	累加 yesShares 和 yesCost。
買 NO	累加 noShares 和 noCost。
查 OPEN	只看還沒結算的持倉。
結算後	持倉狀態從 OPEN 變成 SETTLED，所以 /open 查不到，但全部持倉 API 還查得到。
