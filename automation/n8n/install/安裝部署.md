# UcMarket n8n 環境——安裝部署指南

- 適用：Windows / macOS（Intel 與 Apple Silicon 皆可——映像為多架構 amd64＋arm64，同一份 compose 通用）
- 一句話：裝好 Docker Desktop 之後，一支腳本（或五條指令）把 n8n＋mailpit 帶起來。
- 完全沒碰過 Docker／n8n → 先讀「〇、基礎說明」（3 分鐘）；熟手直接跳「一、安裝步驟」。

---

## 〇、基礎說明（第一次碰的人先讀這節）

### Docker 是什麼

把「程式＋它需要的整個環境」打包成**映像(image)**，在任何機器上以**容器(container)**跑起來，行為一致——把「我這裡可以跑」變成「大家都可以跑」。只要記三個名詞：

| 名詞 | 白話 | 在本專案 |
|---|---|---|
| 映像(image) | 安裝光碟（唯讀範本，可下載、可刪掉重抓） | `n8nio/n8n:2.29.11`、`axllent/mailpit:v1.30.4` |
| 容器(container) | 光碟「跑起來的程式實體」，刪掉重建很便宜 | `n8n-n8n-1`、`n8n-mailpit-1` |
| volume | 外接資料碟——容器刪了，資料還在 | `n8n_data`（n8n 的帳號、workflow、憑證都存這） |

`docker-compose.yml` 則是一張「開機清單」：宣告要跑哪些容器、什麼版本、開哪些 port，`docker compose up -d` 一條指令照單全開。

### n8n 是什麼

開源**自動化平台(workflow automation)**——用拖拉節點的方式把「排程 → 打 API 拿資料 → 判斷 → 發通知」串成一條條 workflow，改流程不用改後端程式。本專案用它做四件事：後端健康告警、每日營運摘要、市場截止提醒（皆發 Discord）、審核通知寄送（後端 worker 打 webhook 進來，n8n 負責寄 email）。

### mailpit 是什麼

**開發用假信箱**：一個假 SMTP 伺服器，開發期所有信都被它攔下來、在網頁上看，**不會真的寄出去**。正式上線時整個移除、換真 SMTP。

### 跑起來的全貌

```text
你的機器
├─ Docker
│   ├─ n8n 容器      ← 瀏覽器開 localhost:5678（workflow 編輯器）
│   │    └─ n8n_data volume（帳號／workflow／憑證）
│   └─ mailpit 容器  ← 瀏覽器開 localhost:8025（收件匣）；n8n 寄信打 mailpit:1025
└─ 後端 Spring Boot（8080，之後的關卡才需要）
     ↑ n8n 容器內呼叫它要寫 host.docker.internal:8080，不是 localhost
```

## 一、安裝步驟（照順序做）

### 步驟 1：裝 Docker Desktop（唯一要手動裝的東西）

- Windows：官網下載 Docker Desktop，需 WSL2 後端（安裝程式會引導）
- macOS：官網下載對應晶片版本（Intel／Apple Silicon）直接裝
- **驗證**：終端機跑 `docker version`——要同時看到 Client 與 Server 兩段版本；只有 Client＝Docker Desktop 沒開

### 步驟 2：啟動環境（兩條路二選一）

先取得專案並進資料夾：`git pull` 後 `cd automation/n8n`（Windows 為 `cd automation\n8n`）——compose 在這層，日常操作也都在這層。

#### 2A. 一鍵腳本（趕路用）

| 平台 | 指令 |
|---|---|
| Windows（PowerShell） | `.\install\setup.ps1`（或對 install\setup.ps1 右鍵以 PowerShell 執行） |
| macOS | `./install/setup.sh`（第一次先 `chmod +x install/setup.sh`） |

腳本做五件事：檢查 Docker → 拉映像（**Docker 原生下載進度條**）→ 起容器 → 等 n8n 就緒 → **全新安裝時**自動匯入 `workflows/` 快照；最後自動開啟兩個網頁並列出後續步驟。失敗會把原因與解法留在畫面上等你按 Enter，不會閃退。
（偵測到既有安裝會**跳過匯入**——避免蓋掉你在 UI 改過但沒匯出的 workflow；想用 git 快照覆蓋本機時才跑 `install\import-workflows.ps1`／`bash install/import-workflows.sh`。）

#### 2B. 手動逐指令（想知道每步在幹嘛的用這條）

這五步涵蓋腳本的主要動作（腳本另外會在全新安裝時自動匯入 `workflows/` 快照——手動版指令見 `workflows/README.md` 匯入節）：

| # | 指令 | 在做什麼 | 你該看到什麼 |
|---|---|---|---|
| 1 | `docker version` | 確認 Docker 引擎活著 | Client 與 Server 各一段版本；只有 Client＝Desktop 沒開 |
| 2 | `docker compose up -d` | 照 compose 拉映像→建網路與 volume→起兩個容器 (寫死讓每個人對齊版本) | 第一次有幾分鐘下載進度條，最後 `Started` ×2 |
| 3 | `docker compose ps` | 列出這個專案的容器 | 兩列，STATUS 都是 `Up` |
| 4 | `curl.exe http://localhost:5678/healthz`（Windows）<br>`curl http://localhost:5678/healthz`（macOS） | 問 n8n 活了沒 | `{"status":"ok"}`（首次啟動要初始化，30 秒內多試幾次） |
| 5 | 瀏覽器開 <http://localhost:5678> 與 <http://localhost:8025> | 進 n8n 編輯器＋mailpit 收件匣 | 5678 出「建立帳號」頁、8025 出空收件匣 |

> Windows 小坑：PowerShell 裡的 `curl` 是 `Invoke-WebRequest` 的別名，輸出長相不同——打 `curl.exe` 才是真 curl（Win10 以上內建）。

### 步驟 3：首次設定（每台機器做一次）

1. 開 <http://localhost:5678> → 建**自己的** owner 帳號（存在本機 volume，不共享、不進 git）
2. 開 <http://localhost:8025> → 確認 mailpit 收件匣打得開
3. workflow 不用手動匯：全新安裝時 setup 腳本已自動匯入 `workflows/` 三條快照（**匯入後一律是停用狀態**）。剩下三步：
   - 左側 Credentials 建三個憑證：`discord-ucmarket-通知`／`discord-ucmarket-心跳`／`smtp-mailpit`（**憑證不隨 JSON 匯出**——Discord webhook URL 向整合線索取；smtp-mailpit 填 Host `mailpit`、Port `1025`、SSL 關、帳密空）
   - 逐條開啟 workflow，在有憑證的節點重新選取剛建的憑證
   - 右上 **Inactive → Active** 逐條啟用；各條用途與驗法見 `workflows/README.md`

### 步驟 4：裝完的驗證清單

- [ ] `docker compose ps` → 兩個容器都是 Up
- [ ] `curl http://localhost:5678/healthz` → `{"status":"ok"}`（Windows 用 `curl.exe`）
- [ ] 瀏覽器開 <http://localhost:8025> 看得到收件匣

全勾＝安裝完成。

## 二、參考（用到再查）

### 日常操作

| 動作 | 指令（在 automation/n8n/ 內） |
|---|---|
| 停 | `docker compose stop` |
| 起 | `docker compose up -d` |
| 看 n8n log | `docker compose logs -f n8n` |
| 徹底重置 | `docker compose down -v` ⚠️ 會刪掉本機的 n8n 帳號、workflow 與憑證 |

不想 cd 進去也行：compose 已用 `name: n8n` 釘死專案名，在任何位置用 `docker compose -f <路徑>/docker-compose.yml <動作>` 都操作同一組容器。

### 版本規則

compose 裡的版本是**刻意釘死**的（n8n `2.29.11`、mailpit `v1.30.4`）——所有人起來的環境一模一樣，禁用 `latest`。
選版原則：**N-1 minor 熟成版**（上一條 minor 線、patch 累積多、仍在維護），不追剛出爐的最新 minor。
升級＝改 compose 一行 → 本機重跑步驟 4 驗證 → commit；誰 pull 誰同步。

### 疑難排解

| 症狀 | 解法 |
|---|---|
| `docker version` 只有 Client | Docker Desktop 沒開，開它 |
| port 5678／8025／1025 被占用 | 本機自行改 compose 冒號**左側**的 port（右側容器內 port 別動）；這種本機改動**不要 commit** |
| workflow 內打後端 API 不通 | 容器內要打 `host.docker.internal:8080`，不是 `localhost:8080` |
| workflow 寄信不通 | SMTP 主機填 `mailpit`、port `1025`（compose 服務名），不是 localhost |
| 開 5678 一直轉圈 | 首次啟動要初始化約 30 秒；`docker compose logs n8n` 出現 `Editor is now accessible` 即可 |
| `git pull` 後版本變了、`up -d` 起不來 | **升版**通常自動換映像重建就好（volume 資料保留、n8n 自動遷移）；**降版必炸**（volume 裡的資料庫 schema 比程式新）——開發資料可拋就 `docker compose down -v` 重置再重跑 setup 腳本（會偵測到全新 volume 自動匯回 workflow；帳號、憑證要重建） |
| setup 顯示「workflow 自動匯入沒成功」 | 環境本身沒壞；等 n8n 完全就緒後跑 `install\import-workflows.ps1`（mac：`bash install/import-workflows.sh`）。重跑安全——同 id 覆蓋，不會長出重複的 workflow |
| 機器上本來就有別的 n8n（其他專案／舊練習）佔 5678 | `docker compose ls` 列出所有 compose 專案揪出誰佔 → 到那個資料夾 `docker compose stop`，或本機改我們 compose 左側 port |

### 安全既定事項

- 所有 port 只綁 `127.0.0.1`——只有本機連得到，同網段其他機器進不來
- 金鑰（Discord webhook URL、token 類）一律存 n8n 憑證庫；`.env` 已被 .gitignore 擋住
- mailpit 只屬於開發期，正式環境會整個移除換真 SMTP
